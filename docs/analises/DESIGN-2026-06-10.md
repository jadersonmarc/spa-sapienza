# Auditoria de Design & Performance — 2026-06-10

> Gerada pela skill `design-review`. Escopo: sistema de estilo
> (`app/globals.css`), componentes de seção, build de produção local
> (commit `bf7c986`) com medições reais via `curl`/`ls`.

## Resumo executivo

A estética dark premium é coerente e bem executada no geral — espaçamento de
seções disciplinado, hierarquia tipográfica correta, efeitos com identidade.
Mas o sistema tem **três rachaduras estruturais**: a classe `font-display`,
usada em todos os títulos importantes, **não existe no tema** (a tipografia
de display planejada nunca renderizou); os efeitos-assinatura
(`.glass`/`.glow`) usam **cores hardcoded fora dos tokens** (se a paleta
mudar, o glow continua violeta); e o site serve **592 KB de foto para um
avatar de 48 px** em toda página de post. Em performance, o peso do código
está dentro do esperado para Next.js, com um CSS de 128 KB inflado pelos 56
componentes shadcn vendorizados sem uso.

## Achados

### [ALTO] `font-display` é uma classe morta — a tipografia de display nunca renderizou
- **Onde**: `app/globals.css` — o `@theme inline` define apenas
  `--font-sans` e `--font-mono`; não existe `--font-display`. A classe
  `font-display` é usada no h1 do hero, títulos de services/portfolio,
  h2 de differentials e no nome da marca no header.
- **Problema**: no Tailwind 4, utility de fonte sem token correspondente
  não gera CSS — todos os títulos caem no `font-sans` silenciosamente.
- **Impacto**: a intenção de design (display type distinto para títulos)
  existe no código mas não na tela; é polimento pago e não entregue.
- **Recomendação**: definir `--font-display` no `@theme` — pode ser a
  própria Geist com peso/tracking próprios (sem nova fonte), ou uma display
  real se a marca decidir (conecta com o brand book, SPEC-25). Alternativa
  mínima: remover as classes mortas.

### [ALTO] Avatar de 592 KB servido em toda página de post
- **Onde**: `public/marc.png` (592 KB) renderizado a 48 px no bloco de
  autor de cada post e 128 px em `/sobre`; `images.unoptimized: true`
  (necessário no VPS) entrega o arquivo inteiro.
- **Impacto**: ~592 KB extras por página de post em mobile — maior custo
  individual de performance do site; candidato a LCP em `/sobre`.
- **Recomendação**: redimensionar para 256 px e comprimir (webp ou png
  ≤ 40 KB). Já coberto pela **SPEC-18** (auditoria UX) — reforça a
  prioridade dela.

### [MÉDIO] Efeitos-assinatura com cores hardcoded fora do sistema de tokens
- **Onde**: `app/globals.css` — `.glass` usa `rgba(20, 20, 30, 0.6)`;
  `.glow-primary` e `.glow-text` usam `rgba(139, 92, 246, …)` (violeta
  fixo), enquanto a paleta oficial vive em oklch (`--primary: oklch(0.65
  0.25 265)`).
- **Problema**: o glow não acompanha o token. A paleta está marcada como
  experimental/não formalizada (branding) — quando ela for decidida, os
  efeitos ficarão dessincronizados e ninguém vai saber por quê.
- **Recomendação**: reescrever com a paleta viva, ex.:
  `box-shadow: 0 0 20px color-mix(in oklch, var(--primary) 30%, transparent)`.

### [MÉDIO] Vocabulário de badges/pills fragmentado (5 estilos)
- **Onde**: hero usa `Badge` shadcn outline; tags do portfólio são `span`
  custom `rounded-full text-[10px]`; pilar do blog é outro `span` custom
  (mesma receita, escrita duas vezes — `components/article-card.tsx` e
  `components/portfolio.tsx`); stack badges são `rounded-md font-mono
  text-xs`; botões do filtro do blog são outra pill custom.
- **Problema**: cinco implementações para o mesmo conceito visual;
  divergências sutis (raio, tamanho, tracking) que o olho percebe como
  falta de acabamento — e manutenção quintuplicada.
- **Recomendação**: um componente `Tag`/`Pill` com variantes (cor,
  tamanho, mono) usado em todas as seções.

### [MÉDIO] CSS de 128 KB inflado por 56 componentes shadcn sem uso
- **Onde**: `components/ui/` tem 56 componentes vendorizados; o site usa
  ~6 (button, card, badge, breadcrumb…). O Tailwind 4 escaneia todos os
  arquivos e gera as classes deles no CSS principal (128,3 KB medidos).
  No `package.json`, dependências órfãs acompanham: `recharts`,
  `react-hook-form`, `embla-carousel-react`, `next-themes` (dark é fixo),
  `react-day-picker`, `vaul` etc., além de `components/theme-provider.tsx`
  sem nenhum import.
- **Impacto**: CSS maior do que o site precisa, instalação lenta e ruído
  de manutenção; `pnpm install` carrega dezenas de pacotes mortos.
- **Recomendação**: remover os componentes de `components/ui/` sem
  import + `pnpm remove` das dependências órfãs (verificar uso com grep
  antes de cada remoção, padrão SPEC-06).

### [BAIXO] `transition-all` generalizado (22 ocorrências na home)
- **Onde**: cards de services/portfolio, footer social, botão WhatsApp.
- **Problema**: anima qualquer propriedade que mude (custo de paint e
  risco de animações não intencionais) quando `transition-colors` ou
  lista explícita bastaria.
- **Recomendação**: trocar por transições específicas nos componentes de
  seção (manter shadcn como está).

### [BAIXO] `overflow-x: hidden` global em `html` e `body`
- **Onde**: `app/globals.css` `@layer base`.
- **Problema**: band-aid clássico — esconde vazamentos horizontais em vez
  de corrigi-los e quebra `position: sticky` de descendentes.
- **Recomendação**: investigar a causa (provável: blur circles do hero com
  `w-96` sem `overflow-hidden` no pai — o hero já tem `overflow-hidden`,
  então testar a remoção do global).

### [BAIXO] CTA do header destoa da geometria pill
- **Onde**: `components/header.tsx` — barra e nav são `rounded-full`, mas
  o botão CTA (shadcn `size="sm"`) é `rounded-md`.
- **Recomendação**: `rounded-full` no botão do header (uma classe).

### Pontos fortes (manter)
- Espaçamento de seções disciplinado (`py-16 sm:py-24` em todas as seções
  da home) e grids alinhados (3 colunas consistentes).
- Hierarquia de headings correta em todas as rotas (h1 único, h2/h3 em
  ordem); `text-balance`/`text-pretty` bem aplicados.
- Fontes via `next/font` com preload e subset latin — 52 KB no total, com
  fallback ajustado (sem CLS de fonte).
- Estados de foco cobertos: shadcn com `focus-visible` e o seletor global
  `outline-ring/50` cobre os elementos custom.
- Filtro do blog com `aria-pressed` e empty state — microinteração com
  acabamento.

## Quick wins

1. Definir `--font-display` no `@theme` — uma linha, destrava a tipografia
   de display em todos os títulos.
2. Comprimir `marc.png` (592 KB → ~40 KB) — maior ganho de performance por
   minuto investido.
3. `rounded-full` no CTA do header — uma classe.
4. Glow com `color-mix(var(--primary))` — três declarações.
